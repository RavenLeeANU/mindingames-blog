# 独立游戏开发设计实践沉思录

# 一、兴起

独立游戏过去一直是我想要尝试的项目，早在大学期间就使用 Unity 做过一些游戏的 Demo，但是碍于那时觉得自己会得太少，无奈去做了游戏策划，最后也没有深入地学习到游戏制作的精髓。目前还是有些不死心，尝试利用业余时间做了一些独立游戏项目，但是都没有完整地把项目做完。但也学到了一些经验，在这里总结和回顾一下，以防后续还不死心想要做游戏时重蹈过去的覆辙。

# 二、立项

项目立项是游戏开发的重中之重，在早期的开发中我并没有意识到立项决定了项目后期的制作周期、开发难度和工作量等等细节。对于独立开发者作者来说，立项首先不能考虑的是宏大的叙事和精美的效果，这些在后期都会导致成本的暴增，以至于根本无法完成游戏。

立项阶段常犯的错误有

**过度脑补：**想到了一个绝无仅有的 idea 后非常兴奋以至于忽略了很多现实的问题和细节。就像你要去画一幅橘子的画，你自动脑补了光影和纹理等的细节，而只关注了画一个圆形，最后导致实际的产出和脑补存在很大的差别。过度脑补也会造成忽略细节实现的工时，导致工作量在实现过程中暴增，永远无法完成。解决过度脑补问题只能通过设计最简游戏机制来达成，比如通过卡片和数值的变化来验证这个 idea 是否值得继续迭代。

**无中生有：**想到了一个全新的玩法，而没有调研市场上的相似产品都是什么样的，导致了对游戏原型的玩法缺乏验证，不成体系。当玩法出现问题后又只能通过修修补补或者增加新的内容来解决，最终导致游戏玩法设计的崩塌。天下没有新鲜事，只要认真调研总能找到相似的游戏，通过观察这些游戏可以帮助获取灵感和找到 idea 中没有考虑到的问题。

**两难问题：**在立项时考虑到成本问题，我们是应该使用现成的工作或资产做一个模仿或设计游戏还是不要屈服于贫穷设计新的创意。在实践中证明了使用现成的内容做命题作文也很困难，尤其是针对独立游戏需要扩展的时候，不可避免地陷入新增还是妥协的过程。但反过来想，如果只是为了做游戏而做游戏，那么立项的意义就不存在了。因此我们可以在前期使用替代的资源，但是不能本末倒置，变成因为没有资源而不去制作这个功能。

**不要小看叙事**：如果游戏中有叙事部分，那就不要给玩家喂屎，糟糕的叙事会毁掉整个游戏。而能够流畅叙事则需要大量的前置工作，这些工作不亚于游戏编程的开发。如果你不擅长叙事，则不要给玩家喂屎，专注于游戏机制本身的设计上，可以增加一些叙事菜单而不要将叙事作为你的卖点。

那么如何保证立项的成功率呢？在没有成为大师之前，我觉得应该先去了解你感兴趣的独立游戏并且从多个维度对游戏进行分析。

# 三、原型

可能会在任何时间点推翻原型而回到立项阶段，即金字塔开发模型。原型期的主要受众是自己和周围的人。但是自己对原型的影响会比较大，主要自己存在先入为主的观念和新鲜感逐渐淡化后的厌恶情绪。如果作为作者都不成持续游玩上百小时，那又能指望谁游玩那么久呢。

原型阶段哪些需要考虑，哪些不需要：

**游戏流程：**独立游戏可能的最佳实践是肉鸽，因为在资源有限的情况下，尽量提高游戏的重复利用程度是十分重要的，在原型期我们不应该关注公关的游戏流程，而是要抓住最核心的玩法流程体验，可能只是一到两个点。然后验证完成这一到两个点的可玩性后，再去扩展整个流程。通常情况下，提前考虑整个长流程的体验会导致无尽的修改和推翻，我们甚至没有机会看到 MVP 版本是什么感觉的一款游戏。

**关卡：**即使游戏有着紧密链接，也需要我们抽象出一些整块的关卡，这方便我们在测试时观察效果，同时也能够孤立地评价去除掉各种限制之后，一个关卡能够称之为好的设计。所以关卡是我们需要优先考虑的，并且关卡能够如何扩展，包含多少种变化也直接决定了我们游戏的整个流程的可玩性。

**美术风格：**我不确定我们是否应该提前拥有美术风格的感受，因为在设计阶段，我们的策划中很有可能已经有了一个最初的游戏的感觉，而这种感觉最早很可能就是来自于某种美术风格或者某个游戏的美术 like。考虑美术风格可以让我们的游戏的风格更加清晰，但是我并不建议太早确定风格，尤其是在原型还会被反复推翻的阶段。另外太好的美术资产加入游戏还会影响我们对游戏可玩性的判断。

**UI 设计：**我们通常在 MVP 时会花费时间在游戏的完成度上，但是这里要清晰一点的是，游戏设计就像画画一样，如果你要画的是一幅肖像画，那么这幅画的完成度绝不取决于任务背后的背景中的某一个柜子的木纹是否真实。也就是说如果我们需要 MVP 的体验完整，那么一套可以在确定位置上点击的 UI 就足够了，那些动效和艺术字什么的完全可以后面再设计；当前你的游戏如果是以 UI 驱动的就另当别论了。

**资产：**我们应该在原型阶段考虑资产的获取难度吗？我觉得不应该，至少不会出现我们要做面包但是没有面粉的情况，一般情况下我们可以把面包做小一点或者去买点面粉，前提是你要做的面包是能吃的。你的游戏也一定是有市场的。

# 四、开发


## 该在何时完成，在何时放弃

开发过程中可能会遇到一些问题，比如出现 MVP 版本比预想得糟糕的情况，或者出现根本无法实现得功能。我们应该何时推翻重做或者干脆放弃这个点子呢。按照过去得失败经历来看，如果是核心 MVP 版本中的功能达不到预期效果，我们需要退回原型阶段看看是否可以改变或者扩展这个玩法让游戏变得有趣。如果不行就需要退回立项阶段，但是这就很伤了，需要记录和复盘下到底出了什么问题。

如果是核心功能无法实现，则需要评估究竟是确实无法通过现有技术实现还是自己不会。多数情况下应该是自己不会，而学习会额外花费时间成本。如果核心玩法依赖这个功能，那么花点时间是有价值的。如果是表现功能，那可以先往后做，慢慢来或者寻求替代方案。注意这里是替代方案，而不是妥协的方案。

开发量激增导致没有信心做下去也是常有的事情，这在过去已经体会到无数次了。但是放弃一个做了一半的游戏本身就是一种极大的浪费，你不知道你的游戏是否会火，连面世的机会都没有了。所以最好还是能够充满信心地做下去。信息的来源有二，一是充分的市场调查，另一个就是需要自己的盲目自信。除此之外没人能够帮到你。

## 注意内容边界

目标不确切的市场调查通常会带来新的问题，比如又看上了别人的新功能，决定增加新的系统和内容等。这时又会投入新的成本进来开发新功能，然而老的功能都没有得到验证。因此，我们不能狗熊掰苞米，我们需要目的明确地验证某个事情。调研和验证的目标或者问题需要非常清晰，不能单纯地问这个游戏好玩吗之类的，我们需要更加聚焦，比如在建造系统中，在摆放建筑时弹出建筑物的属性列表对于用户选择是否有帮助这样的问题。看看大部分具备类似功能的游戏是如何设计的，除此之外别看人家的优点。不要做三心二意的人。

## 应该准备什么文档
如果我们自己开发且对开发内容了如指掌，那是没必要记录什么。但是大多数情况下我们需要负责不同的系统，来回在不同的职责间切换，这时我们会忽略或者忘记之前想到的东西，准备一个简要的开发文档有助于我们在切换角色时保持意见的一致性。

UI: 简单的 UI 布局，帮助我们规划跳转和 Layout 的逻辑；

数据结构：需要一个最简的数据结构的表或者文档，用于规划常用的对象的数据类型，便于下一步去配置表格。

UML 图：开发时需要梳理思路，如果我们不能画清楚 UML 图，说明哪里还是有点问题的，需要再次审视设计问题。

策划案和 PR: 我觉得作为独立开发者，可写可不写。如果我们需要一个队员加入，向他讲清楚游戏是最直接的，这些宣传策划案和通常不会起到太大作用，尤其是对于核心规则无法明确描述的段落，等于白给。

# 五、上线

说实话我没有太多的上线经验，暂未踩过后续的坑，但是从我学到的经验来看。即使是再好的游戏，还是需要提前预热和发布试玩版本来提升曝光量。这不仅可以提前触达目标玩家，也可以给自己一定的改错和调整空间。

## 该如何面对暴死
成功需要一定的运气，有时候暴死才是正常的结果。但是大多数情况下正常人没有意志去接受多次的暴死。所以最好的办法还是坚持，并且将曾经暴死的游戏遗骸上吸收经验和知识，并且准备浴火重生。
